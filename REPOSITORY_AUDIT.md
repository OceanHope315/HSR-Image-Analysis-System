本系统只能用于软件开发与功能演示，不能声称具备真实安检识别能力，不能替代安检人员。

## 10. 明显问题

- 多个 Python 脚本硬编码 `D:\Users\...`、`D:\python\...` 等个人路径，换机后无法运行。
- 训练、数据清洗、推理、气体通信混在脚本级全局流程中，不适合作为 Web 后端直接复用。
- 气体模拟器监听 `0.0.0.0:502`；502 是低端口，WSL/Linux 可能需要额外权限，并会暴露到网络接口。
- TCP 读取依赖分包/粘包处理，现有代码的协议解析与 UI 耦合较重；需要通过适配器隔离。
- `runs` 中存在相似的气体/视频演示代码，重复逻辑较多。
- 气体模拟器的解除报警帧长度和 CRC 覆盖范围不一致，`send` 也可能发生部分发送；真实接入前必须先用协议测试向量修复。
- 参考标签定义了 7 类目标：工具、刀具、压力容器、剪刀、充电宝、瓶子、打火机；该顺序将作为未来模型映射参考。
- 大量训练产物、标签和权重不应进入 Web 仓库或 Docker 构建上下文。
- 初始状态没有认证、校验、错误处理、日志、测试、构建或部署基线。
- 因没有现有 Web 代码，不存在可确认的循环依赖；后续用分层和导入检查预防。

- `yoloAdapterService` 当前输出模拟结果，未来通过配置切换到 Python YOLO HTTP 服务，不直接加载外部 `.pt`。
